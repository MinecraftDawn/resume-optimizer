# Optimization Suggestions Reference

## Section Suggestion Format

每個**達成率低於 75%** 的區塊，至少提供 2 條具體建議。格式如下：

```
### [區塊名稱] — X/滿分Y（達成率 Z%）  ⚠️/❌

**問題：** [具體弱點，引用履歷原文]

**建議 1：** [具體行動]
**建議 2：** [具體行動]
[XYZ 改寫示範（工作經歷/專案成就必附）]
```

---

## Achievement Rewrite: Google XYZ Formula

工作經歷與專案成就的每條任務型描述，套用 XYZ 公式改寫：

> **Accomplished [X] as measured by [Y], by doing [Z]**
> 中文版：「透過 [Z 方法]，達成 [X 成果]，以 [Y 指標] 衡量」

**改寫流程：**
1. 識別原句是「任務型」（負責 / 參與 / 協助）還是「成果型」（提升 / 降低 / 達成）
2. 任務型 → 詢問用戶：「這件事帶來什麼具體成果？有沒有數字？」
3. 若用戶提供數字 → 直接套入 XYZ 公式輸出改寫版
4. 若用戶無法提供數字 → 用「估算範圍」替代，並標注 `[待補數字]`

**改寫範例：**

| 原句（任務型） | 改寫後（XYZ 成果型） |
|--------------|-------------------|
| 負責社群媒體管理 | 管理 FB/IG 雙平台內容策略，3 個月自然觸及提升 40%、追蹤人數成長 2,000+，透過發文頻率優化與 UGC 活動執行 |
| 協助行銷預算規劃 | 優化月均 $50k 行銷預算配置，CPA 降低 15% 同時導流量成長 20%，透過 A/B 測試調整渠道比例達成 |
| 參與客戶服務流程改善 | 推動客服 SOP 重新設計，首次解決率從 62% 提升至 81%，透過客訴分類分析找出 Top 5 重複問題並建立標準回覆庫 |
| 負責後端 API 開發 | 設計並交付 RESTful API 服務，支撐日均 50 萬次請求，回應時間從 420ms 降至 95ms，透過 Redis 快取層與查詢優化實現 |

## JD Keyword Gap Analysis

**觸發條件：** 用戶在 Step 1 提供了目標職缺連結或 JD 文字。

### 執行流程

**Step A — 提取 JD 關鍵字**

從 JD 中萃取三類關鍵字：

1. **硬技能關鍵字**（工具、語言、平台、框架）
   - 例：Python、Google Analytics、Salesforce、React、SQL

2. **職能關鍵字**（職責動詞 + 職能名詞）
   - 例：「跨部門協作」、「數據驅動決策」、「客戶成功管理」、「A/B 測試」

3. **產業/域名關鍵字**（產業術語、職位特定詞）
   - 例：SaaS、B2B、電商、敏捷開發、GDPR、KYC

**Step B — 與履歷交叉比對**

對每個 JD 關鍵字，判斷履歷中的狀態：

| 狀態 | 定義 | 標記 |
|------|------|------|
| 完全匹配 | 履歷中有相同或同義詞，且有具體佐證 | ✅ |
| 弱匹配 | 履歷提到但缺乏具體化或語境不同 | ⚠️ |
| 缺失 | 履歷中完全未出現 | ❌ |

**Step C — 輸出 Gap 分析表**

```
### JD 關鍵字 Gap 分析

**目標職缺：** [職缺名稱]

| 類別 | JD 關鍵字 | 履歷狀態 | 建議行動 |
|------|----------|---------|---------|
| 硬技能 | Python | ✅ 完全匹配（工作經歷第2條） | — |
| 硬技能 | Tableau | ❌ 缺失 | 若有使用經驗，加入專長區塊 |
| 職能 | 跨部門協作 | ⚠️ 弱匹配（自傳提及但無佐證） | 在工作經歷中加入「帶領X人跨部門...」描述 |
| 產業 | SaaS | ❌ 缺失 | 若有 SaaS 產品經驗，在工作內容中補充 |

**匹配率：** X / Y 個關鍵字完全匹配（XX%）

**優先補強：** [列出最重要的 3 個缺失關鍵字，及建議放置位置]
```

**Step D — 調整工作經歷 JD 維度分數**

Gap 分析完成後，回到 `scoring.md` 的工作經歷評分，根據實際關鍵字匹配率更新「JD 關鍵字匹配」子項分數：

| 匹配率 | JD 維度分數（/10） |
|--------|-----------------|
| ≥80% | 9–10 |
| 60–79% | 7–8 |
| 40–59% | 5–6 |
| 20–39% | 3–4 |
| <20% | 0–2 |

## ATS 相容性檢查（台灣市場）

**觸發條件：** 用戶目標職缺為中大型企業（員工 200 人以上）、外商、或科技公司。

### 台灣常見 ATS 系統
- 104 履歷直接投遞：104 自有 ATS，使用 104 標準格式影響最小
- 企業自建/第三方 ATS（如 Greenhouse、Lever、Workable）：需注意格式相容
- 外商全球 ATS（如 Workday、SAP SuccessFactors）：對 PDF 格式最嚴格

### ATS 相容性 Checklist

依照以下清單檢查候選人履歷（若用戶提供 PDF，評估格式結構；若只有文字，跳過排版類項目）：

**✅ 建議做（ATS 友善）：**
- [ ] 使用標準區塊標題（工作經歷、學歷、技能，而非創意命名）
- [ ] 每個職位都有明確的公司名稱、職稱、起始時間
- [ ] 技能關鍵字直接出現在描述文字中（不只在標籤欄）
- [ ] 日期格式統一（如 2022/03–2024/06，而非「前年至去年」）

**❌ 避免（ATS 不友善）：**
- [ ] 表格或多欄位排版（ATS 通常無法正確解析）
- [ ] 重要資訊放在頁首頁尾（ATS 常忽略 header/footer）
- [ ] 圖片或圖形式技能評分（如圓形進度條）
- [ ] 嵌入式字型或特殊符號取代文字

**評分影響：** ATS 問題不計入現有評分表，但在「下一步行動清單」中列為優先提醒項，若有嚴重 ATS 問題則在報告開頭的總覽中特別標注。

## Common High-Impact Optimizations

### 個人特色 — 從通用到精準

- Weak: 積極、負責、細心
- Strong: React.js、數據分析、敏捷開發、B2B 業務、英文簡報

### 求職條件 — 期望工作內容策略

有明確 JD → 針對 JD 寫，展現匹配與貢獻：
- ✅ 「期望以數據驅動的行銷策略協助品牌成長，擅長 SEO/內容行銷，目標協助公司提升 organic traffic」

無明確 JD → 留空（保持機會廣度）或寫能力導向內容；避免通用願望清單：
- ✅ 留空：開放策略，不自我設限（合理選擇）
- ✅ 「擅長以 SEO/內容行銷驅動自然流量成長，能協助品牌在 organic 渠道建立長期競爭優勢」
- ❌ 「希望從事行銷相關工作」— 自我設限卻無任何價值

### 自傳開場 — 避免常見模板

- ❌ 「我叫XXX，畢業於OO大學...」
- ❌ 「我是一個積極進取、勇於挑戰的人...」
- ✅ 直接切入最強成就：「在過去 3 年的電商行銷工作中，我主導了 3 次品牌重新定位，每次都帶來雙位數的營收成長...」

### 語文能力 — 可信度建立

- ❌ 中文：精通、英文：精通、日文：精通（無任何佐證）
- ✅ 英文：工作應用（TOEIC 860，前職以英文撰寫所有技術文件）

### 專長可信度 — 交叉驗證

若宣稱某專長，確保工作經歷或專案中有至少一條具體使用記錄，否則：
- 移除該專長，或
- 在工作經歷中補充「使用 [工具/技能] 完成 [具體任務]」

## 自訂內容轉化模板

當用戶確認有特殊成就時，依類型提供對應的 104 自訂內容撰寫格式：

### 學術發表
```
標題：[論文題目（縮短版）]
活動期間：[投稿/發表年份]
說明：
在 [期刊/研討會名稱]（[影響係數或等級，如 SCI, SSCI, EI, 國際研討會]）發表「[論文標題]」，
[核心研究貢獻一句話]。共同作者：[第幾作者，若第一作者特別標明]。
連結：[DOI 或 Google Scholar 連結]
```

### 開源貢獻
```
標題：[專案名稱] 開源貢獻
活動期間：[起始年份–至今]
說明：
貢獻於 [GitHub 專案名稱]（[Stars 數量] stars），[貢獻類型：新增功能/修復 bug/文件]，
影響 [使用者數量或下載量] 用戶。主要 PR：[PR 簡述]。
連結：[GitHub profile 或具體 PR 連結]
```

### 演講 / 社群分享
```
標題：[演講主題]
活動期間：[演講日期]
說明：
受邀於 [活動名稱（如 MOPCON、g0v 黑客松、JavaConf Taiwan）] 發表「[演講標題]」，
現場觀眾約 [人數] 人，[演講重點一句話]。
連結：[投影片 SlideShare/Speaker Deck 或活動錄影]
```

### 重大獎項
```
標題：[獎項名稱]
活動期間：[得獎年份]
說明：
於 [主辦單位] 舉辦的 [活動名稱] 中獲得 [名次/獎項等級]，
[競賽規模：參賽隊伍數/參賽人數]，[成果或解決的問題一句話]。
連結：[得獎公告或作品連結，若有]
```

## 各模式 Step 3 建議策略

**⚠️ 這些規則覆蓋 Step 3 的行動優先序，不改變個別建議的生成方式。**

---

### Mode A（主動求職 有JD）

**優先序：** JD Gap > 工作經歷 > 專長 > 其他

- Step 3 開頭必須輸出 JD Gap 分析（見 JD Keyword Gap Analysis 區塊）
- 建議清單優先列「能在 2 小時內完成且提升 JD 匹配率」的改動
- ATS 問題若存在，列於行動清單第 1 條（比任何內容改善都緊急）
- 自訂內容詢問：必做（技術職尤其重要）

---

### Mode B（主動求職 無JD）

**優先序：** 工作經歷可搜尋度 > 專長廣度 > 個人特色標籤 > 基本資料

**核心目標：讓履歷在多種職位搜尋中都能被找到。**

- **跳過 JD Gap 分析**（無 JD）
- Step 3 開場白使用：「由於目前沒有特定 JD，建議優化方向是提升履歷的廣泛可搜尋度，讓不同公司的 HR 在搜尋時都能找到你。」
- 技能標籤建議：優先建議補充「業界主流關鍵字」，而非針對特定職缺的關鍵字。例："建議在工作技能標籤中加入 [Python/SQL/Excel]，這三個詞在資料相關職位搜尋量最高。"
- 求職條件建議：期望工作內容欄可留空或寫能力導向內容（見上方求職條件策略），不建議過度限定職位類型
- 自傳建議：第一句改為職能亮點，不要綁定特定公司（因為還沒決定投哪裡）
- 自訂內容詢問：必做（沒有 JD 匹配優勢時，差異化更重要）
- ATS 相容性：**跳過**（不知道目標企業規模，不強迫 ATS 優化）

---

### Mode C（被動觀望）

**優先序：** 高槓桿改善 > 其他（避免大幅重寫建議）

- Step 3 開場白：「以觀望模式為前提，以下建議集中在 ROI 最高的改善，目標是讓你的履歷在有機會出現時站得住腳，而不是要你今天就做大改動。」
- 每條建議附上「預估分數提升幅度」與「預估所需時間」（例："補充 2 個工作技能標籤 → 預估 +1 分，約 10 分鐘"）
- **最多提供 5 條建議**，不列長清單（被動觀望者執行意願低，清單太長反效果）
- 若有 JD 則做 Gap 分析，但說明「等你確定目標再針對性調整即可」
- 自訂內容詢問：選做（若分數達成率已 ≥75% 則跳過此提問）
- ATS 相容性：**跳過**

---

### Mode D（職涯轉換）

**優先序：** 可遷移技能表述 > 自訂內容 > 求職條件 > 自傳

- Step 3 開場白：「轉職成功的關鍵不是「有幾年相關經驗」，而是讓對方看到你的技能可以遷移。以下建議聚焦在如何用你已有的背景，說服新領域的面試官。」
- 工作經歷建議：針對每份工作，找出「在新產業/新職能中也有價值的技能」並建議強化描述
- 自訂內容詢問：**必做且優先**（Side project、社群參與、轉職相關課程認證都計入）
- 若自訂內容缺失，特別提示：「轉職時，有沒有在新領域做過任何實作（課程專案、volunteer、freelance）？就算小規模也可以放，這對轉職來說比一般求職者的同等內容更有說服力。」
- 自傳建議：必須包含「轉職動機 + 技能延續性」，提供具體的自傳開場改寫範例

---

### Mode E（晉升/加薪）

**優先序：** 工作經歷成果量化 > 管理責任描述 > 自傳晉升敘事 > 自訂內容

- **跳過 JD Gap 分析**（目標是同公司晉升或業界同職位跳槽，非應徵特定外部 JD）
- Step 3 開場白：「晉升和加薪說服的是同一件事：你的貢獻規模和影響力已超過現在的職等。以下建議集中在讓這件事從履歷上就清楚可見。」
- 工作經歷建議：
  - 每份工作都要問：「管理幾人？管理多少預算？影響哪些部門？」→ 補入描述中
  - 成就描述優先改寫為 XYZ 格式，特別強調規模（"影響 X 人/X 元/X% 的指標"）
- 自傳建議：必須包含「晉升/加薪理由敘事」——我做了什麼、帶來什麼規模的成果、憑什麼值得更高職位/薪資。提供範例開場：
  ```
  ✅ 範例開場：「在過去 2 年擔任資深工程師期間，我主導了 [X 系統] 的重構，
  使團隊的 deploy 時間從 45 分鐘降至 8 分鐘，直接支撐 3 個產品線的快速迭代，
  這也是我認為自己已準備好承擔 Tech Lead 角色的具體依據。」
  ```
- 自訂內容詢問：必做（演講、開源、獎項在晉升場景特別有說服力）
- ATS 相容性：**跳過**（晉升通常不過 ATS）
